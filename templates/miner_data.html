How to use this table
  • Click a period label, Type, or Date cell — opens the Document Viewer as a fixed bottom panel (source text + extraction analysis). Click the X or press Escape to close it.
  • Double-click a numeric value cell — opens an inline editor directly in the cell. Type a number and press Enter or click OK. Pressing Escape restores the original value.
  • Hover over a value cell — browser tooltip shows the raw source snippet that produced the value.
  • Method badges next to values show extraction source: R = regex, L = LLM, A = analyst, F = finalized, D = derived, S = SEC filing, P = pipeline.
  • Columns button — toggle individual metric columns on/off.

How extraction works
Two-stage pipeline
Stage 1 — Ingest: fetch raw document and store text in the reports table. Stage 2 — Extract: run regex and LLM passes against stored text, then route results through the agreement engine.
Document routing
Press releases and archive files (PDF/HTML) run the monthly path. EDGAR 10-Q and 10-K filings run the quarterly/annual path. Each path uses the same extraction stack but different period-granularity expectations.
Regex extraction
Pattern files in config/patterns/ are matched against the full document text. Each match yields a value, a confidence score (pattern weight x distance factor x range factor), and the surrounding context window.
LLM extraction
A single Ollama batch call sends all active metrics at once. The model returns a JSON object with value, confidence, source_snippet, and period_granularity for each metric.
Period granularity filter
Monthly documents reject LLM responses marked as quarterly or annual. Example: LLM returns Q3 total = 1,252 for a September press release — the response is discarded because September data (341) was expected. The regex result is used or the item goes to review.
Agreement engine
If regex and LLM agree within the per-metric threshold, the value is auto-accepted into data_points. If they disagree, or only the LLM found a value, the item is routed to review_queue for analyst decision.
Analyst protection
Rows with extraction_method in (analyst, analyst_approved, review_approved, review_edited) are never overwritten by a pipeline re-run.
Finalization overlay
The final_data_points table is an analyst-controlled override layer. "Finalize" writes here. The data display prioritises final_data_points over raw data_points. "Reprompt LLM" reruns extraction and updates raw data_points only — finalized values are untouched.
